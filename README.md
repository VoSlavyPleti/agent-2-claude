Есть много способов структурировать проект, но наилучшая структура - это структура, которая является последовательной, понятной и не содержит неожиданностей.

Если просмотр структуры проекта не дает вам представления о том, что представляет собой проект, то структура может быть неясной.
Если вам приходится открывать пакеты, чтобы понять, какие модули в них расположены, то ваша структура неясна.

Если расположение модуля и его название не дают вам представления о том, что у него внутри, значит, у вас очень плохая структура.

1. Основные положения и структура
Код вашего бизнес-сервиса хранится в директории src. Директория aigw-project - это непосредственно директория клонирования проекта с bitbucket.

При разработке бизнес-сервиса вам следует использовать собственное название. Это касается директории ** src/aigw_ct** и настроек в разделе [tool.poetry], поля name и packages.

Данная структура является рекомендательной, вы вправе распоряжаться ей по вашему смотрению. Однако если вы решили придерживаться основы предоставленной нами структуры, учитывайте, что некоторые файлы лучше не редактировать самостоятельно, например, src/aigw_ct/api/os_router.py. Такие файлы и места помечены соответствующим комментарием. Не забывайте также удалять ненужный функционал (например, демонстрационный endpoint в src/aigw_ct/api/v1/router.py).

Мы всегда готовы оказаться поддержку и консультировать по структуре и работе данного шаблона. Базовая работоспособность шаблона проверяется при каждых вносимых нами правках. При возникновении ошибок в первую очередь проверяйте логику работы вашего кода и его интеграцию в структуру шаблона, подробно дебажьте и пишите и запускайте тесты 😊

Храните все доменные директории в src
src/aigw_ct - самый высокий уровень приложения, содержит конфигурации, константы и т.д.
__main__.py - корень проекта, который запускает приложение FastAPI
config.py - здесь обрабатываются переменные окружения и другие
base.py - базовые классы для работы приложения
context.py - имплементация паттерна AppContext для управления жизненным циклом компонентов приложения. Основная идея паттерна AppContext заключается в том, чтобы создать специальный объект, который содержит ссылки на различные сервисы, ресурсы и настройки, которые могут быть доступны из любой части приложения. Этот объект создается единожды при запуске приложения и передается через методы или в конструкторы других объектов, где он может быть использован. Однако стоит понимать, что глобальный доступ таит в себе опасности. Поэтому к публичным ресурсам класса следует реализовывать открытые функции, которые будут выступать зависимостями к методам вашего rest приложения
aigw_ct/logger - реализация логгера приложения
context_vars.py - контекстные переменные для сквозного логирования. Приходящие в запросе заголовки сохраняются здесь и используются на протяжении обработки запроса. Механизм контекстных переменных позволяет асинхронно обрабатывать несколько запросов и не смешивать их основные параметры (например, Request-Id) Редактировать нежелательно
logger.py - непосредственно реализация класса логгера. Редактировать нежелательно
models.py - модели для объекта логирования, метрик, события аудита. Редактировать запрещено
utils.py- вспомогательные функции для логгера и настройка маскирования.
uvicorn_logging_config.py- конфигурация встроенного в uvicorn логгера.
aigw_ct/api - уровень имплементации бизнес-логики
__init__.py - конфигуратор FastAPI приложения. Здесь настраивается версионирование api, а также какие методы должны быть включены в релиз
os_router.py - глобальные инфраструктурные методы: health check и info. Редактировать запрещено
metric_router.py - глобальные инфраструктурные методы для сбора метрик. Редактировать запрещено
schemas.py- pydantic схемы для инфраструктурных методов. Редактировать запрещено
middleware.py- логика перехвата FastAPI сообщений. Запись дефолтных логов, событий аудита, метрик Редактировать запрещено
aigw_ct/api/v1 - версия бизнес логики
router.py - является ядром co всеми endpoint-ми(ручками)
schemas.py - pydantic модели для описания ответов, тел запросов, возвращаемых ошибок и т.д.
models.py - модели для работы с бд, дата классами, именованными кортежами и т.д.
service.py - бизнес-логика, специфичная для конкретного сервиса
utils.py - функции, не связанные с бизнес-логикой, например, нормализация ответов, обогащение данных и т.д.
exceptions.py - исключения, относящиеся к конкретному сервису. PostNotFound, InvalidUserData и т.д.

ПРИ ЛОКАЛЬНОЙ РАЗРАБОТКЕ С СЕРТИФИКАТАМИ:
Обратите отдельное внимание на переменную окружения LOCAL. Она предназначена для разграничения порядка подключения к другим АС локально и на стендах. Если вам нужно отправлять запросы в стороннюю АС, то локально вам понадобятся соответствующие сертификаты, которые вы подставляете в коде. На стендах (в том числе на ИФТ) сертификаты в коде подставлять не нужно, это делается автоматически на стадии egress (выхода трафика из pod). Соединение на стендах не установится, если вы также будете подставлять сертификаты и в коде. Для разграничения и нужна переменная LOCAL. Вы можете реализовывать это на свой вкус, мы лишь предупреждаем о такой потребности и предлагаем разграничивать через булеву переменную LOCAL.

Далее в коде, например, вы делаете класс-интерфейс для общения со сторонней АС и можете разделить логику примерно следующим образом (используется библиотека aiohttp и самый простой вариант разделения):

def __init__(self, logger, cert: str, key: str, cacert: str, local: bool):
    self._logger = logger
    self._ssl_ctx = False
    self._local = local
    if local:
        self._logger.debug('App working outside of OS with local certificates')
        if not (cert and key and cacert):
            self.logger.warning('No certificates provided')
        self._ssl_ctx = ssl.create_default_context(cafile=cacert)
        self._ssl_ctx.load_cert_chain(certfile=cert, keyfile=key)
        self._ssl_ctx.check_hostname = False
        self._ssl_ctx.verify_mode = ssl.CERT_NONE
    self._logger.debug('App working inside OS without certificates')
    self._session = None

3. Журналирование
Подробно журналирование описано в документации на confluence. За основу мы используем loguru, о нем вы можете дополнительно почитать здесь.

В данном шаблоне реализовано базовое логирование (оно же журналирование): есть объект логгера, который хранится в контексте приложения (src/aigw_ct/context.py). Вы можете импортировать его в любое место приложения, а также уже реализовано логирование входящих и исходящих через FastApi запросов. Подробнее вы можете ознакомиться с этой логикой в src/aigw_ct/api/middleware.py, где используется "перехват" запросов до их попадания в основной код приложения и непосредственно после окончания обработки запроса.

Логи пишутся в консоль и в файл (в переменных окружения вы как раз задавали путь до него). Из этого файла на стенде fluent-bit будет вычитывать логи и отправлять их в АС Тенгри, а вы сможете в UI эти логи посмотреть.

НИ В КОЕМ СЛУЧАЕ НЕЛЬЗЯ ПИСАТЬ ИНФОРМАЦИЮ УРОВНЯ К2 И ВЫШЕ В ЛОГИ
Если требуется писать сообщения в логи, часть полей которых содержат чувствительную информацию, обратитесь к src/aigw_ct/logger/utils.py. Там есть функция для маскирования и там же настраивается само маскирование. Все тела входящих и исходящих сообщений (в поле args.message) по умолчанию маскируются полностью, через конфиг в том же файле вы можете изменить это, но перед этим ознакомьтесь со страницей про маскирование в разделе журналирования на confluence.

По большей части от вас требуется только реализовать логирование вашего собственного кода. Делать это вы можете на свое усмотрение, однако чем подробнее, яснее и последовательнее вы это сделаете, тем легче потом будет смотреть логи и отлавливать ошибки.

Пример из в src/aigw_ct/api/v1/router.py (внутри вызываемых методов, разумеется, тоже надо делать логирование) :


logger.info("Start processing request")
try:
    _gc_resp = await service.business_logic(gc_session=gc_session,
                                            gc_settings=request.gc_settings,
                                            logger=logger,
                                            inn=request.inn, ogrn=request.ogrn)
    gc_answer = _gc_resp.choices[0].message.content
except grpc.RpcError as rpc_error:
    logger.error(f"Request failed: {rpc_error.details()}", exception=repr(rpc_error))
    return JSONResponse(status_code=424,
                        content=schemas.FailedDependencyResponse(code=str(rpc_error.code()),
                                                                 error_description=str(
                                                                     rpc_error.details())).model_dump())
logger.info("End processing request")


Таким образом, все, что вам нужно для того, чтобы сделать лог, это у объекта логгера вызвать метод, соответствующий уровню логирования, и передать в него само сообщение. Оно будет записано в поле message верхнего уровня схемы логирования. Какой уровень в каких случаях выбирать опять же на ваш вкус и опыт, напомним лишь, что основных 7

Level name	Severity value	Logger method
TRACE	5	logger.trace()
DEBUG	10	logger.debug()
INFO	20	logger.info()
SUCCESS	25	logger.success()
WARNING	30	logger.warning()
ERROR	40	logger.error()
CRITICAL	50	logger.critical()
В примере кода выше также показано, как заполнять поле stackTrace при ошибке. Все остальные поля заполняет за вас логгер в данном шаблоне, в том числе поле args при обработке входящих и исходящих сообщений.

Обратите внимание, что в консоли также фигурируют логи, выделенные красным цветом. Это логи uvicorn, они не пишутся в файл и не будут фигурировать в UI просмотра логов. Логи uvicorn нужны для отладки и настраиваются через src/aigw_ct/logger/uvicorn_logging_config.py. Настройки затем прописываются в src/aigw_ct/__main__.py. При желании запись этих логов в консоль можно отключить.

Кроме того, силами того же логгера реализовано отбрасывание событий аудита и метрик (метрики на момент написания находятся в проработке, но механизм уже заложен). Об аудите и метриках подробнее читайте ниже.

4. Аудит
Подробно аудит описан в документации на confluence.

Аудит, по сути, требует записывать целиком тела входящих и исходящих сообщений в специальную систему. Это не для вас, а для безопасников и только на стендах. В шаблоне это уже реализовано "из коробки" в том же месте в middleware, где входящие и исходящие сообщения пишутся в логи. Вам достаточно убедиться, что при запуске приложения и отправке в него сообщений у вас появляется файл по локации, определенной в переменных окружения, и что в этот файл пишутся сообщения.

5. <BETA> Метрики
Подробно метрики описаны в документации на confluence.

Метрики, как и логи, пишутся в АС Тенгри. Ваша задача состоит в заполнении файла, путь до которого прописан в переменных окружения. Логгер уже "из коробки" поддерживает запись метрик по методу .metric()

logger.metric(
    metric_name="ai_tc_service_requests_total", 
    metric_value=1,
)
"Из коробки" уже пишется основная часть метрик, также реализованы пути для оценки бизнес-сервиса и записью соответсвующей метрики.

Концепция прорабатывается, данный раздел будет дополняться. Пока что вам не требуется самостоятельно реализовывать ничего.

6. Инструкции
SberOSC poetry
Официальная документация poetry

Документация по настройке SberOSC

Установите токен аутентификации. Получить можно в sberosc в разделе профиль -> перевыпустить токен:

poetry config http-basic.sberosc token {sberosc-token}
Сертификат SSL verification:

curl -kv https://nexus.sigma.sbrf.ru/nexus/content/repositories/OAIR_tool_release/cert/cacert_sigma.pem -o {ваша директория для сертификатов. рекомендуем ~/.certs}/sberca-chain.pem
poetry config certificates.sberosc.cert ~/.certs/sberca-chain.pem
При ошибке сертификата [SSL: CERTIFICATE_VERIFY_FAILED] нужно отключить проверку:

poetry config certificates.sberosc.cert false
Установить зависимости при сформированном poetry.lock файле можно с помощью команды

poetry install
При внесении изменений в файл pyproject.toml выполните

poetry update
Локальный запуск
В зависимости от способа установки python один из вариантов запуска может быть следующая команда:

python3 src/aigw_ct/__main__.py
Если вы пользуетесь IDE PyCharm, то запускать можно по нажатию на зеленый треугольник, находясь в указанном файле.

После успешного запуска станет доступен swagger бизнес-сервиса по адресу http://0.0.0.0:8080/docs. Хост и порт могут меняться в зависимости от переменных окружения и вашей платформы. На Windows дополнительно проверьте адрес http://127.0.0.1:8080/docs

Docker

Запуск линтеров
Конфигурации линтеров описаны в pyproject.toml, в секции [tool.ruff] и [tool.pylint.*] Все команды над src выполнять из корневой директории клонирования (rest_template по умолчанию)

pylint src
Pylint должен набирать более 7 баллов! Обратите внимание, что "из коробки" pylint дает больше 9 баллов.

ruff src
Для удобства, можно использовать установленные в зависимостях isort и black. isort позвоялет организовать импорты в правильном порядке, а black выполняет автоматические форматирование кода. Обратите внимание, что black по умолчанию содержимое списков, параметры, аргументы и др. организует однострочно, если вы хотите многострочно, то добавьте замыкающую запятую (trailing comma) в конце соответствующего объекта.

isort src
black src

8. Сборка дистрибутива
От мастер ветки создаем ветку release/<номер_релиза>
от ветки release/<номер_релиза> создаем story/(RGSUDP | RGSCT)-<номер_story>
в стори черипикаем задачи из develop
делаем PR из story в release/<номер_релиза>
